# autoLV 요구사항 (v5)

## 1. 목표
다음 방식으로 개별공시지가를 조회하는 웹 서비스를 제공해야 한다.
- 지번 기반 단건 조회
- 도로명 기반 단건 조회
- (다음 단계) 엑셀 업로드 기반 대량 조회(최대 10,000행)

## 2. 사용자 유형
- 비로그인 사용자: 단건 조회 가능
- 로그인 사용자: 단건 조회 + 조회기록 확인
- 관리자: 계정만 준비됨(기능 확장 예정)

## 3. 기능 요구사항
### 3.1 주소 선택 UI
- 지번 검색은 `시/도 → 시/군/구 → 읍/면/동` 선택 후 지번을 입력해야 한다.
- 도로명 검색은 `시/도 → 시/군/구 → 자음 → 도로명` 선택 후 건물번호를 입력해야 한다.
- 자음 목록은 해당 시/군/구에 실제 도로명이 있는 자음만 보여줘야 한다.

### 3.2 주소 정규화
- 지번 검색은 `ld_code + is_san + main_no + sub_no`로 PNU를 생성해야 한다.
- 도로명 검색은 좌표 변환/역변환을 통해 지번을 찾고 PNU를 생성해야 한다.

### 3.3 공시지가 조회
- VWorld 개별공시지가 API를 통해 실데이터를 조회해야 한다.
- 같은 연도 데이터가 여러 건이면 최신 갱신본을 우선해야 한다.
- 조회 결과는 기준년도 내림차순으로 보여줘야 한다.

### 3.4 인증
- 이메일/비밀번호 회원가입/로그인/로그아웃을 제공해야 한다.
- 인증은 HttpOnly 쿠키(access_token/refresh_token) 기반으로 처리해야 한다.
- 회원가입 검증 규칙(이메일 중복, 비밀번호 정책, 닉네임 형식, 약관 동의)을 적용해야 한다.

### 3.5 조회기록
- 로그인 사용자의 단건 조회 결과를 최신순으로 표시해야 한다.
- 기록 클릭 시 해당 조회 결과를 다시 열람할 수 있어야 한다.
- 현재 저장소는 클라이언트(localStorage) 기반이며 서버 영구 저장은 다음 단계로 분리한다.

### 3.6 파일 조회(준비 단계)
- 파일조회 페이지/상태 UI는 존재해야 한다.
- 실제 업로드/비동기 처리/다운로드 API는 아직 미구현이며 다음 단계 요구사항으로 유지한다.

### 3.7 지도 조회 (2단계 확장)
- 지도 기반 조회는 Kakao Maps JS SDK를 사용해야 한다.
- 사용자가 지도 클릭 시 아래 순서로 데이터를 계산해야 한다.
1. 클릭 좌표(lat/lng) 획득
2. VWorld API로 좌표를 PNU로 변환
3. 공공데이터 API로 당해연도 개별공시지가 조회
4. 전년도 개별공시지가 조회
5. 필지 면적 조회
6. 아래 계산값 생성
   - 현재 공시지가
   - 전년 대비 증감률(YoY)
   - `면적 × 단가`
   - 반경 200m 인근 평균가(PostGIS `ST_DWithin`)
- 계산 결과는 지도 우측(또는 하단) 결과 패널에 표시해야 한다.

### 3.8 지도 조회 성능/중복 방지 (2단계 확장)
- 지도 클릭 이벤트는 300ms 디바운스를 적용해야 한다.
- 동일 클릭/동일 PNU에 대한 중복 API 호출을 방지해야 한다.
- PNU 변환 결과는 Redis에 24시간 TTL로 캐시해야 한다.
- 공시지가 API 결과는 서버 DB(PostgreSQL)에 캐시해야 한다.

### 3.9 지도 조회 데이터 저장 (2단계 확장)
- PostgreSQL + PostGIS를 도입해야 한다.
- `parcels` 테이블을 사용해야 한다.
  - `id` (PK)
  - `pnu` (UNIQUE)
  - `lat`
  - `lng`
  - `area`
  - `price_current`
  - `price_previous`
  - `updated_at`
- 반경 조회를 위해 PostGIS `geography` 컬럼을 유지해야 한다.

### 3.10 CSV Export (2단계 확장)
- `GET /export?pnu=...` 엔드포인트를 제공해야 한다.
- CSV 컬럼은 아래 순서를 유지해야 한다.
  - `pnu`
  - `area`
  - `current_price`
  - `previous_price`
  - `growth_rate`

### 3.11 지도 조회 보안 (2단계 확장)
- 외부 API 키는 `.env`로만 관리하고 클라이언트에 노출하면 안 된다.
- 외부 API 호출은 서버(FastAPI)에서만 수행해야 한다.

### 3.12 공간 집계(폴리곤) 조회 (2단계 확장)
- 사용자는 Kakao Map에서 여러 점을 연결해 폴리곤을 그릴 수 있어야 한다.
- 프론트는 폴리곤 좌표 배열을 백엔드로 전송해야 한다.
- 백엔드는 좌표를 WKT `POLYGON((...))` 형식으로 변환해야 한다.
- 백엔드는 PostGIS `ST_Intersects` 기반 집계를 수행해야 한다.
```sql
SELECT
  COUNT(*) as parcel_count,
  SUM(price_current) as total_price,
  AVG(price_current) as avg_price,
  SUM(area) as total_area
FROM parcels
WHERE ST_Intersects(
  geom,
  ST_GeomFromText(:polygon, 4326)
);
```
- 집계 결과(`parcel_count`, `total_price`, `avg_price`, `total_area`)를 반환해야 한다.

### 3.13 공간 집계 DB 확장 (2단계 확장)
- `parcels` 테이블에 공간 컬럼을 추가해야 한다.
```sql
ALTER TABLE parcels
ADD COLUMN geom GEOMETRY(POLYGON, 4326);

CREATE INDEX idx_geom_gist
ON parcels
USING GIST (geom);
```
- 공간 집계/반경 조회는 공간 인덱스를 활용해야 한다.

### 3.14 공간 집계 성능/안전 제약 (2단계 확장)
- 폴리곤 최대 꼭짓점 수는 100개로 제한해야 한다.
- 폴리곤 면적이 5km²를 초과하면 요청을 거부해야 한다.
- 집계 쿼리에는 타임아웃을 설정해야 한다.
- 동일 요청에 대한 중복 호출 방지 로직을 적용해야 한다.

### 3.15 공간 집계 API (2단계 확장)
- `POST /aggregate` 엔드포인트를 제공해야 한다.
- 입력: 폴리곤 좌표 배열(위도/경도)
- 출력: 집계 결과와 사용된 폴리곤 메타데이터(면적, 꼭짓점 수)

### 3.16 지도 확장 구현 산출물 (2단계)
- FastAPI `/aggregate` 엔드포인트 구현
- WKT 변환 유틸리티 구현
- Kakao Map 폴리곤 드로잉/전송 예시 코드
- PostGIS 공간 집계 쿼리 예시
- Redis/DB 캐시 연계 설계 문서

## 4. 비기능 요구사항
- 보안: 비밀번호 해시, JWT 만료, 환경변수 기반 키 관리
- 안정성: 외부 API 장애/키 오류 시 명확한 오류 메시지 반환
- 사용성: 모바일 대응 레이아웃과 명확한 상태 메시지 제공
- 성능: 도로명 텍스트 파일 조회는 캐시를 사용해 반복 조회 성능을 보장해야 한다
- 지도 확장 시 성능: 반경 조회는 PostGIS 인덱스를 사용해야 하며 클릭 지연을 최소화해야 한다.
- 공간 집계 확장 시 성능: 대형 폴리곤 차단, 쿼리 타임아웃, 인덱스 강제 사용을 보장해야 한다.

## 5. 범위 제외 (현재 릴리스)
- 엑셀 10,000행 비동기 처리
- 소셜 로그인
- 서버 DB 기반 조회기록/통계 대시보드
- 지도 조회(2단계 확장 요구사항으로 정의됨)

## 6. 인수 기준 (현재 릴리스)
- 지번/도로명 단건 조회가 VWorld 실데이터로 동작해야 한다.
- 로그인/로그아웃/회원가입/내 정보 조회가 정상 동작해야 한다.
- 조회기록에서 주소가 PNU가 아닌 사람이 읽는 주소 형식으로 보여야 한다.
- 문서(요구사항/아키텍처/API/DB)가 실제 구현과 일치해야 한다.

## 7. 인수 기준 (2단계: 지도 조회)
- 지도 클릭 시 PNU 변환 -> 공시지가 조회 -> 계산 패널 표시가 완료되어야 한다.
- 200m 인근 평균가가 PostGIS 반경 조회로 계산되어야 한다.
- Redis 24시간 캐시가 적용되어 동일 PNU 재조회 시 응답이 빨라져야 한다.
- `GET /export?pnu=...`가 유효한 CSV 파일을 반환해야 한다.
- 폴리곤 집계 요청 시 `/aggregate`가 집계 결과를 정상 반환해야 한다.
- 꼭짓점 100개 초과/면적 5km² 초과 폴리곤은 검증 오류로 거부되어야 한다.
- 공간 집계 쿼리 타임아웃이 설정되어 장시간 락/대기 상황을 방지해야 한다.
